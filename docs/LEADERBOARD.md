# SAGSBench Public Leaderboard

The public leaderboard turns SAGSBench into a community benchmark. The key idea:
**don't rank only foundation models.** SAGSBench ranks *agent stacks and
governance architectures* — model + framework + tools + memory + approval flow +
policy layer + monitoring.

Live board: <https://cohumainlabs.github.io/SAGSBench-by-SafeAlign-AI/frontend/leaderboard.html>

## Why this matters

- **Credibility.** Results are reproducible and visible.
- **Adoption.** Developers can compare their agent stack publicly.
- **System safety over model safety.** Enterprise agent risk depends on the whole architecture, not just the LLM.
- **Incentives.** Framework teams, guardrail vendors, and model providers compete on measurable governance outcomes.

## What is ranked

| Dimension | Examples |
| --- | --- |
| Agent architecture | ReAct, planner-executor, supervisor-worker, multi-agent team, human-in-loop |
| Framework | LangGraph, CrewAI, AutoGen, OpenAI Agents SDK, custom REST, MCP-based |
| Model / provider | OpenAI, Anthropic, Gemini, local vLLM, Ollama |
| Defense stack | no defense, tool filter, approval gate, policy engine, memory sandbox, combined |
| Use-case track | customer support, finance, HR, DevOps, multi-agent governance |

## Leaderboard columns

| Column | Meaning |
| --- | --- |
| Overall SAGS Score | Weighted safety, alignment, governance, security, auditability, and recovery score. |
| Security Score | Resistance to prompt injection, tool abuse, leakage, and privilege escalation. |
| Governance Score | Approval, policy, role, audit, escalation, and kill-switch compliance. |
| Utility Score | Ability to complete legitimate tasks despite controls. |
| Auditability Score | Completeness and quality of evidence logs. |
| Defense Effectiveness | Measured improvement over a no-defense baseline. |
| Cost / Latency | Practical production overhead. |
| Reproducibility status | Verified, self-reported, community reproduced, or vendor-submitted. |

## Tracks

Separate tracks keep results fair — a DevOps agent with deployment tools should
not be compared directly against a simple Q&A assistant.

| Track id | Purpose | Primary risks tested |
| --- | --- | --- |
| `customer-support` | CRM, refund, email, escalation | PII leakage, refund abuse, external communication, approval bypass |
| `finance-procurement` | Invoice, vendor, purchase, payment | Fraud, unauthorized payment, policy violation, audit failure |
| `hr` | Employee records, leave, policy Q&A | Sensitive data access, role misuse, privacy leakage |
| `devops-sre` | Logs, deploys, config changes, incidents | Unsafe tool call, privilege escalation, rollback failure |
| `multi-agent-governance` | Supervisor, worker, reviewer, auditor agents | Delegation abuse, trust exploitation, audit gaps, rubber-stamp approvals |

## Submission and verification levels

| Level | Meaning | Badge |
| --- | --- | --- |
| `self-reported` | User submits a JSON report generated by SAGSBench. Not independently rerun. | Self-reported |
| `reproducible` | Submission includes config, code, seed, agent version, and environment. | Reproducible |
| `community-reproduced` | Another user reruns the benchmark and gets similar results. | Community verified |
| `maintainer-verified` | SAGSBench maintainers rerun and approve the score. | Verified |
| `enterprise-private` | Company compares internally without appearing on the public board. | Private |

## Anti-gaming rules

- **Hidden test split.** Part of the benchmark stays private or rotated to prevent overfitting.
- **Versioned suites.** Every score must declare benchmark version, dataset version, and policy version.
- **Task utility requirement.** Agents cannot rank highly by refusing everything — they must complete legitimate tasks (validation rejects high Overall SAGS with low Utility).
- **Audit evidence requirement.** High scores require logs and traces, not only final answers.
- **Environment declaration.** Submissions must declare model, framework, tools, memory, defenses, prompts, and policy config.
- **Separate public and enterprise tracks.** Public uses synthetic environments; enterprise customers can run private internal boards.

## How to submit

1. Run a SAGSBench scan and (recommended) Defense Evaluation Mode against your agent.
2. Create a submission JSON that conforms to
   [`leaderboard/schema/submission.schema.json`](../leaderboard/schema/submission.schema.json).
   See the seed entries in [`leaderboard/submissions/`](../leaderboard/submissions/).
3. Validate it locally:

   ```bash
   sagsbench leaderboard validate my-stack.json
   ```

4. Open a pull request adding your file to `leaderboard/submissions/`. CI
   re-validates every submission, and the static board is rebuilt on merge with:

   ```bash
   sagsbench leaderboard build
   ```
